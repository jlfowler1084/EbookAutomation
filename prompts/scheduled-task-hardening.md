## Context

`Install-EbookScheduledTask` exists (PSM1 line ~3046) and registers a Windows Scheduled Task that polls the inbox every 15 minutes. It has **never been tested end-to-end**. The supporting functions `Uninstall-EbookScheduledTask` and `Get-EbookTaskStatus` also exist.

Issues found during code review:

1. The auto-generated `Run-Pipeline.ps1` launcher calls `Invoke-EbookPipeline` with **no parameters** — doesn't respect user-defaults.json for GenerateMP3, UseClaudeChapters, etc.
2. No error handling in the launcher — if the pipeline throws, the task silently fails
3. No retry logic — a transient failure means that book is skipped until manually retried
4. No daily summary — no way to see what was processed without reading log files
5. No log rotation awareness — logs grow indefinitely

## Task 1: Harden the Launcher Script

The auto-generated `Run-Pipeline.ps1` (written by `Install-EbookScheduledTask`) needs to be more robust. Replace the current 2-line launcher with:

```powershell
# Auto-generated launcher for EbookAutomation scheduled task
# Regenerated by Install-EbookScheduledTask — do not edit manually
$ErrorActionPreference = 'Continue'

try {
    Import-Module '<MODULE_PATH>' -Force

    # Read user defaults for pipeline flags
    $defaults = Get-EbookDefaults

    $pipelineArgs = @{}
    if ($defaults.generate_mp3)       { $pipelineArgs['GenerateMP3'] = $true }
    if ($defaults.use_claude_chapters) { $pipelineArgs['UseClaudeChapters'] = $true }

    Invoke-EbookPipeline @pipelineArgs
}
catch {
    # Write error to a persistent location (not just console)
    $errorLog = Join-Path '<PROJECT_ROOT>' 'logs' 'scheduler-errors.txt'
    $timestamp = Get-Date -Format 'yyyy-MM-dd HH:mm:ss'
    "$timestamp  SCHEDULER ERROR: $_" | Add-Content $errorLog -Encoding UTF8
    "$timestamp  Stack: $($_.ScriptStackTrace)" | Add-Content $errorLog -Encoding UTF8
    exit 1
}
```

Where `<MODULE_PATH>` and `<PROJECT_ROOT>` are replaced with the actual resolved paths at generation time (same as current code).

### user-defaults.json integration

`Get-EbookDefaults` returns a hashtable. Check for these keys and pass them through:
- `generate_mp3` → `-GenerateMP3`
- `use_claude_chapters` → `-UseClaudeChapters`

If `Get-EbookDefaults` returns a default object (file doesn't exist), the pipeline runs with default behavior — no change from current.

## Task 2: Add Error Retry to Invoke-EbookPipeline

In `Invoke-EbookPipeline`, when a book fails BOTH TTS and Kindle conversion, the file is left in the inbox (current behavior is correct). But the processed manifest marks it as seen, so it won't be retried.

Add retry awareness:
1. When a book fails all conversion steps, do NOT add it to the processed manifest (`Add-ProcessedFile`). Currently the processed manifest is only updated on success (line ~2956: `if ($anySuccess)`), so verify this is actually working correctly.
2. Add a failure counter: track how many times a file has failed in a `logs\failed-books.json` file. Schema: `{"filename.pdf": {"failures": 2, "last_attempt": "2026-03-29T...", "last_error": "..."}}`.
3. After 3 failures for the same file, log a warning and skip it (so a permanently broken PDF doesn't block the queue every 15 minutes).
4. Add a `-RetryFailed` switch to `Invoke-EbookPipeline` that resets failure counts and retries all previously-failed books.
